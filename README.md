# ai-codeshift

Spec将取代code变为唯一可信源（Single Source of Truth），零行手写从根本上改变了软件工程，围绕编程相关的实践将逐渐淡出我们的视线，Spec工程才是未来。

1. 渐进式披露：`AGENTS`的重要性，定义项目的大边界，约定俗成的规范。
2. 上下文工程：上下文完整性与上下文隔离性既是控制大模型幻觉的重要原则，也是保障功能代码删除后可依据规范重新准确生成的关键基础。
3. 架构风格：是AI适应架构还是架构顺应AI，值得探索。
4. 代码在各角色和阶段要做知识转化，消耗大，衰减大。语义化知识传播失真更少，可复用性更高。

## 为何围绕编码的实践将不再重要？
回顾以往的编码实践，诸如各种架构风格(DDD、MVC)、TDD、整洁代码、SOLD、职责边界、无重复代码等，它们的共性目的都是为了使人理解和修改代码的成本更低，在AI时代，编写、重构、理解代码都可以靠AI解决，这些实践当然就不再重要了，但并不是说完全取代，只是重心转移。
例如代码审查也许没有Spec审查重要，但同样有价值。

## Spec工程重点
1. Spec is Truth：Spec并不仅是驱动代码生成，而是要完全取代代码，简而言之，Spec因包含所有业务知识。文档和代码冲突时，错的一定是代码。
2. No Spec, No Code：没有文档，不准写代码。任务的修改都应该从Spec开始，再根据Spec重新生成代码，切勿人为修改代码。
3. 任何功能删除代码后都能够根据Spec重新生成。

## 衍生出新的软件工程

围绕Spec会衍生出新的软件工程实践：
1. Spec工程师。
2. 基于业务生成Spec。
3. Spec审查。
4. 基于Spec的构建、打包、测试、上线。
5. 基于Spec的测试案例生成。
6. 基于Spec的变更影响分析。
7. .....

Harness Engineering

```
specs
├── AGENTS.md                         # 项目简介、项目目录结构、技术栈、功能代码生成规范
├── features                          # 业务领域集
│   └── feature                       # 业务领域
│       ├── entity
│       │   └── entity.md             # 数据库表契约
│       └── functions                 # 业务功能集
│           └── function              # 业务功能，原子化
│               ├── api.md            # 业务功能API契约
│               ├── service.md        # 业务逻辑
│               ├── mapper.md         # 数据库操作
│               └── feginclient.md    # 第三方API契约
└── scaffold                          # 项目脚手架
    └── scaffold.md
```
